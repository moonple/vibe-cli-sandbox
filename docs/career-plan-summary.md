# 产品1（Vibe Coding-inspired）12 周交付计划（AI PM 平台型）

**起始日期**：2026-03-23  
**结束日期**：2026-06-14  
**定位**：开发者场景 AI 应用，但重点展示平台能力：**Eval-first + Observability-first + 回归迭代**  
**硬件/约束**：WSL2 + Ubuntu + RTX 3060 6GB（Hugging Face 访问受限）

---

## 产品1：最终验收（到 2026-06-14）
- ✅ 可演示 demo：输入任务 → 输出方案/patch → 本地验证 → 生成报告
- ✅ 有 Trace：每次请求可追踪阶段耗时与错误
- ✅ 有 Eval：至少 30 条任务用例集 + 回归对比报告
- ✅ README 可复现：别人按步骤能跑、能看到指标

---

## Week 1（2026-03-23 ~ 03-29）：MVP 骨架 + 跑通闭环
### 目标
把“能跑模型”变成“产品最小闭环”。

### 任务
- [ ] 定义 1-2 条用户故事（谁、要做什么、成功标准）
- [ ] 定义最小输入/输出格式（建议：Markdown 模板或 JSON）
- [ ] 做出端到端 demo（允许手动验证，但流程必须通）

### 产出
- `docs/demo-script.md`（3 分钟演示脚本）
- `docs/output-schema.md`（输出结构定义）

### 验收标准
- 能录屏一次完整流程（不要求美观，要求闭环）

---

## Week 2（2026-03-30 ~ 04-05）：输出结构化 + 错误分类
### 目标
让输出可评测、可追踪。

### 任务
- [ ] 输出结构固定：plan / patch(or diff) / commands / risks / fallback
- [ ] 定义失败类型（格式错误/执行失败/超时/拒答/不相关）

### 产出
- `docs/error-taxonomy.md`

### 验收标准
- 任意 5 条任务输出结构一致且可读

---

## Week 3（2026-04-06 ~ 04-12）：Observability v1（日志 + 指标）
### 目标
能回答“慢在哪里/失败在哪里”。

### 任务
- [ ] request_id
- [ ] 阶段耗时：total /（可选 prefill、decode）
- [ ] token 统计：prompt_tokens / gen_tokens（能拿到就记录）
- [ ] 失败原因写入日志

### 产出
- `docs/observability.md`
- `results/trace_samples.md`（≥5条）

### 验收标准
- 能用日志定位一次失败原因

---

## Week 4（2026-04-13 ~ 04-19）：Eval v0（30 条用例）
### 目标
建立“可重复运行的评测”。

### 任务
- [ ] 设计 30 条 coding 任务用例（修 bug/写函数/写测试/重构）
- [ ] 定义评分方法（可先人工/半自动）
- [ ] 输出基线报告

### 产出
- `eval/cases_v0.md`
- `eval/report_v0.md`

### 验收标准
- 同一批用例能复跑并生成对比表

---

## Week 5（2026-04-20 ~ 04-26）：质量改进 v1（用 Eval 驱动）
### 目标
从“感觉更好”变成“指标更好”。

### 任务
- [ ] 统计失败 Top3
- [ ] 针对 Top3 改 prompt/策略
- [ ] 做 v0 vs v1 对比报告

### 产出
- `eval/report_v1.md`（含对比表）

### 验收标准
- 成功率或关键失败类别明显改善（给出数字）

---

## Week 6（2026-04-27 ~ 05-03）：可靠性 v1（超时/取消/降级）
### 目标
让 demo 更像真实可用工具。

### 任务
- [ ] 超时与取消策略
- [ ] 降级策略（缩短 ctx / 降低输出长度 / 简化任务）
- [ ] 失败时给“可执行下一步”

### 产出
- `docs/reliability.md`

### 验收标准
- 失败时不“沉默”，能给出明确恢复路径

---

## Week 7（2026-05-04 ~ 05-10）：性能与成本叙事（配置对比）
### 目标
建立平台型 PM 的“质量/成本/延迟”取舍能力。

### 任务
- [ ] 固化性能采集（TTFT/tokens/s）
- [ ] 记录至少 3 组配置对比（ctx/batch/并发或线程）
- [ ] 写一页取舍结论

### 产出
- `results/perf_compare.md`

### 验收标准
- 能讲清楚一次“为什么选这个配置”

---

## Week 8（2026-05-11 ~ 05-17）：看板化展示（Markdown 报表也行）
### 目标
面试官一眼看懂：你优化了什么、数据怎么变。

### 任务
- [ ] 汇总核心指标表（成功率、TTFT、tokens/s、失败Top）
- [ ] 固化报告模板

### 产出
- `results/dashboard.md`

### 验收标准
- 仓库首页/结果页能 10 秒理解成果

---

## Week 9（2026-05-18 ~ 05-24）：Eval 扩展到 60~100 + 回归流程
### 目标
让迭代具备“平台味道”：规模化、可回归。

### 任务
- [ ] 扩充用例集（≥60）
- [ ] 引入版本标签（prompt v1/v2、策略版本）
- [ ] 回归运行脚本化（不强求全自动评分）

### 产出
- `eval/cases_v1.md`
- `eval/report_v2.md`

### 验收标准
- 能稳定输出“版本对比报告”

---

## Week 10（2026-05-25 ~ 05-31）：PM 叙事文档（产品故事）
### 目标
把项目翻译成 PM 语言。

### 任务
- [ ] 写清楚：用户是谁、痛点是什么、怎么验证、指标怎么选
- [ ] 写清楚：失败与迭代决策

### 产出
- `docs/product-story.md`

### 验收标准
- 一页讲清“做了什么+为什么+数据证明”

---

## Week 11（2026-06-01 ~ 06-07）：最终 Demo + README 打磨
### 目标
形成对外展示版本。

### 任务
- [ ] 录制 3-5 分钟 demo
- [ ] README 首页优化（10 秒读懂）
- [ ] README 放上关键指标与截图

### 产出
- `README.md`（最终版）
- `docs/demo.md`

### 验收标准
- Demo 连续演示 3 次不翻车

---

## Week 12（2026-06-08 ~ 06-14）：复盘 + 产品2立项（PRD）
### 目标
把产品1封装成作品集资产，并启动产品2。

### 任务
- [ ] 产品1复盘（做得好/不足/下一步）
- [ ] 产品2 PRD 草稿（目标用户、MVP、指标）

### 产出
- `docs/postmortem.md`
- `docs/prd-product2.md`

### 验收标准
- 产品2边界清晰，MVP 可在 4-8 周落地
