# Reliability

本文件整理 `vibe-cli-sandbox` 中需要关注的可靠性策略，主要参考 `career-llm-plan` 中的产品规划，以及 `llm-infer-deploy-lab` 中的推理部署经验。

## 1. 目标

让 demo 不只是“能跑”，而是“失败时也能给出明确恢复路径”。

## 2. 需要覆盖的可靠性能力

### 超时
- 为每次任务设置超时
- 超时后返回清晰错误
- 不允许静默挂起

### 取消
- 支持用户主动取消
- 取消后清理中间状态
- 保留必要日志用于排查

### 降级
- 缩短 ctx
- 降低输出长度
- 简化任务
- 在资源紧张时优先保证可用性

### 重试
- 对短暂网络或执行失败做有限次重试
- 避免无限重试
- 重试需保留原因记录

### 并发控制
- 控制同时处理的任务数
- 避免卡死与雪崩
- 6GB 显存环境下尤其要保守

## 3. 常见故障处理

### OOM
- 减少 `--n-gpu-layers`
- 降低 `-c`
- 使用更低量化
- 检查模型是否放在 WSL 本地

### 输出异常
- 检查 prompt 是否过长
- 检查输出结构是否符合 schema
- 检查 mock/backend 是否返回了空内容

### 执行失败
- 记录失败类型
- 给出下一步建议
- 确保用户能继续执行而不是卡住

## 4. 评价标准

一个可靠的版本应该满足：
- 出错时有明确错误信息
- 出错时有恢复建议
- 不会无限等待
- 不会因为单点失败导致整个流程中断

## 5. 对当前仓库的建议

后续实现时，建议在 CLI 和 core 层都保留：
- timeout 参数
- cancellation hook
- fallback / recovery 字段
- error type 分类
